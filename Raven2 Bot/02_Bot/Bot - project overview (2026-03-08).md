Я продолжаю долгосрочный проект бота для анализа рынка террамондов в MMORPG Raven 2. Проект уже частично реализован: у меня есть инфраструктура на Linux‑сервере, Git‑репозитории и базовый backend, который записывает историю аукциона в SQLite через FastAPI/SQLAlchemy. Я хочу, чтобы ты подхватил этот контекст и помогал мне дальше развивать бота и аналитику.

## 1. Цели проекта (подробно)

**Главная цель:** создать полуавтоматического/автоматического бота‑аналитика, который зарабатывает на колебаниях цен террамондов в Raven 2, при этом максимально снижая риск бана и давая мне удобную аналитику по рынку.

Бот должен уметь:

1. Регулярно собирать данные с аукциона террамондов:
    
    - периодически делать «снимки» аукциона (через скриншоты/захват окна клиента Raven 2);
        
    - вытаскивать из них список лотов: тип террамонда, цена за штуку, количество, направление сделки (buy/sell) и время;
        
    - сохранять всё это в локальную БД, чтобы накопить историю цен и объёмов.
        
2. Строить аналитику и выявлять паттерны:
    
    - считать базовую статистику по каждому террамонду: средняя цена, минимум/максимум, объёмы торгов;
        
    - анализировать изменения по дням недели и времени суток (например, «по будням днём цены ниже, вечером выше»);
        
    - искать повторяющиеся паттерны: окна, когда обычно выгодно закупаться, и окна, когда обычно выгодно продавать;
        
    - настраиваемые стратегии: фильтры по типу террамонда, минимальная маржа, лимиты по объёму, ограничения по риску и т.п.
        
3. Работать в двух режимах:
    
    - **Режим ассистента**: бот только выдаёт сигналы, что сейчас выгодно покупать/продавать.
        
        - Примеры сигналов: «сейчас цена wood_basic ниже средней на 20%, окно для закупки», «цена rare_terra выше 90‑го перцентиля, можно продавать».
            
        - Вывод сигналов в лог, файл, потом можно прикрутить уведомления в Telegram/Discord.
            
    - **Авто‑режим**: бот сам имитирует действия игрока в клиенте.
        
        - Открывает аукцион, находит нужный террамонд.
            
        - Покупает/выставляет лоты по рассчитанной стратегии.
            
        - Делает это с имитацией человеческого поведения: случайные задержки, движения мышью, перерывы, лимит по длительности сессий, чтобы снижать палевность.
            
4. Инструменты для контроля и отладки:
    
    - просмотр содержимого БД: последние лоты, распределение цен, базовые отчёты;
        
    - логи действий бота: какие лоты куплены/проданы, какие сигналы сгенерированы, по каким причинам;
        
    - возможность в любой момент выключить авто‑режим и оставить только подсказки.
        

Важный момент: игра не предоставляет официального API, поэтому всё приходится делать через экран/клиент, OCR и автокликер, а значит надо особо внимательно относиться к безопасности аккаунта, тестировать сначала на твинке и избегать слишком «идеального» поведения бота.

## 3. Текущая структура backend‑проекта `raven2-bot`

Репозиторий `raven2-bot` расположен в `/home/bmv/projects/raven2-bot` и сейчас выглядит так (логическая структура):

- `backend/`
    
    - `app/`
        
        - `__init__.py`
            
        - `db.py` — настройка SQLAlchemy:
            
            - `DATABASE_URL = "sqlite:///./data/raven2.sqlite"`
                
            - `engine = create_engine(...)`
                
            - `SessionLocal = sessionmaker(...)`
                
            - `Base = declarative_base()`
                
            - `get_db()` — зависимость FastAPI для выдачи сессий.
                
        - `models.py` — ORM‑модели:
            
            - `Terramond`:
                
                - таблица `terramonds`;
                    
                - поля: `id` (PK), `code` (уникальный код террамонда), `name` (читаемое имя);
                    
                - связь `lots` → список лотов данного террамонда.
                    
            - `Snapshot`:
                
                - таблица `snapshots`;
                    
                - поля: `id`, `taken_at` (время снимка, по умолчанию `datetime.utcnow`), `source` (идентификатор источника — VM/клиент);
                    
                - связь `lots` → список лотов, вошедших в этот снимок.
                    
            - `Lot`:
                
                - таблица `lots`;
                    
                - поля: `id`, `snapshot_id` (FK → `snapshots.id`), `terramond_id` (FK → `terramonds.id`), `unit_price` (цена за единицу), `quantity` (количество), `direction` (строка `buy` или `sell`);
                    
                - связи: `snapshot` (обратно к `Snapshot`), `terramond` (обратно к `Terramond`).
                    
        - `schemas.py` — Pydantic‑схемы для входящих данных:
            
            - `LotIn`:
                
                - `terramond_code: str`
                    
                - `terramond_name: str`
                    
                - `unit_price: float`
                    
                - `quantity: int`
                    
                - `direction: str` (`"buy"` / `"sell"`)
                    
            - `SnapshotIn`:
                
                - `taken_at: datetime` — время снимка в UTC;
                    
                - `source: str | None` — идентификатор источника снимка;
                    
                - `lots: list[LotIn]` — список лотов в этом снимке.
                    
        - `crud.py` — логика работы с БД:
            
            - `get_or_create_terramond(db, code, name)`:
                
                - ищет террамонд по `code`, если нет — создаёт, коммитит и возвращает;
                    
            - `create_snapshot_with_lots(db, snapshot_in)`:
                
                - создаёт запись в таблице `snapshots` с `taken_at` и `source`;
                    
                - для каждого лота в `snapshot_in.lots`:
                    
                    - вызывает `get_or_create_terramond`;
                        
                    - создаёт запись в `lots` с правильными FK, ценой, количеством и направлением;
                        
                - коммитит изменения и возвращает объект `Snapshot`.
                    
        - `main.py` — FastAPI‑приложение:
            
            - создаёт приложение `app = FastAPI()`;
                
            - на событие `startup` вызывает `Base.metadata.create_all(bind=engine)`, чтобы гарантировать наличие таблиц;
                
            - реализует эндпоинт:
                
                - `POST /snapshots`:
                    
                    - принимает JSON по схеме `SnapshotIn`;
                        
                    - вызывает `crud.create_snapshot_with_lots`;
                        
                    - возвращает `{ "snapshot_id": <id> }`.
                        
- `docker/`
    
    - `Dockerfile.backend` — Dockerfile для backend‑сервиса:
        
        - базовый образ: `python:3.11-slim`;
            
        - устанавливает зависимости: `fastapi`, `uvicorn[standard]`, `sqlalchemy`, `pydantic`;
            
        - `WORKDIR /app`;
            
        - `COPY backend ./backend`, `COPY data ./data`;
            
        - открывает порт `8000`;
            
        - команда запуска: `uvicorn backend.app.main:app --host 0.0.0.0 --port 8000`.
            
- `data/`
    
    - папка для SQLite БД (`raven2.sqlite`), которая монтируется внутрь контейнера;
        
    - БД создаётся автоматически при первом запуске через `Base.metadata.create_all()`.
        
- `docker-compose.yml`
    
    - сервис `backend`:
        
        - `build.context` — текущая директория;
            
        - `build.dockerfile` — `docker/Dockerfile.backend`;
            
        - `container_name: raven2-backend`;
            
        - `volumes`:
            
            - `./backend:/app/backend` — чтобы можно было править код на хосте и видеть изменения;
                
            - `./data:/app/data` — чтобы SQLite лежала на хосте и сохранялась между рестартами;
                
        - `ports`:
            
            - `"8000:8000"` — доступ к API на хосте по `http://localhost:8000`;
                
        - `restart: unless-stopped`.
            

## 4. Что уже сделано функционально

С точки зрения функционала, на данный момент:

1. **База данных:**
    
    - определены три основные сущности:
        
        - `Terramond` (справочник террамондов по коду);
            
        - `Snapshot` (один момент времени снятия аукциона);
            
        - `Lot` (конкретный лот в рамках снимка).
            
    - обеспечена связь many‑to‑one:
        
        - много `Lot` → один `Snapshot`;
            
        - много `Lot` → один `Terramond`.
            
2. **API для приёма снимков аукциона:**
    
    - реализован REST‑эндпоинт `POST /snapshots`;
        
    - он принимает JSON следующей структуры:
        
        - `taken_at` — время снимка в UTC;
            
        - `source` — строковый идентификатор источника (`"manual_test"`, `"win_vm1_client1"` и т.п.);
            
        - `lots` — массив объектов с полями:
            
            - `terramond_code`
                
            - `terramond_name`
                
            - `unit_price`
                
            - `quantity`
                
            - `direction` (`"buy"` / `"sell"`).
                
    - при вызове эндпоинта:
        
        - создаётся запись в `snapshots`;
            
        - для каждого лота:
            
            - гарантируется наличие записи `Teramond` с указанным `code`;
                
            - создаётся запись в `lots` с правильным `snapshot_id` и `terramond_id`;
                
        - в ответ возвращается `{"snapshot_id": <id>}`.
            
3. **Docker‑обёртка:**
    
    - собран контейнер FastAPI‑backend’а с SQLite;
        
    - docker‑compose позволяет поднять сервис одной командой;
        
    - БД живёт вне контейнера, чтобы при пересборке не терять данные.
        
4. **Проверка работоспособности:**
    
    - после запуска `docker compose up -d`:
        
        - по адресу `http://localhost:8000/docs` открывается Swagger UI FastAPI;
            
        - при тестовом запросе:
            
            json
            
            `{   "taken_at": "2026-03-08T10:00:00Z",  "source": "manual_test",  "lots": [    {      "terramond_code": "wood_basic",      "terramond_name": "Basic Wood",      "unit_price": 123.45,      "quantity": 100,      "direction": "sell"    }  ] }`
            
            API возвращает `{"snapshot_id": 1}`, а в SQLite появляются соответствующие записи в `terramonds`, `snapshots` и `lots`.
            

По сути, сейчас у меня есть минимальный, но работающий «приёмник данных» с аукциона, который позволяет постепенно накапливать историю по всем террамондам.
