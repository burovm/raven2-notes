# Windows‑клиент Raven 2 — архитектура

## Цель Windows‑слоя

Windows‑слой — это «тонкий клиент», который работает на Windows‑машине/VM с установленным клиентом Raven 2 и:

- запускает/контролирует игру;
    
- периодически делает снимки аукциона;
    
- парсит их (OCR → структура лотов);
    
- формирует JSON по контракту `POST /snapshots`;
    
- отправляет данные на Linux‑backend `raven2-bot` (`http://192.168.50.185:8000`);
    
- опционально запрашивает сигналы (`/signals/...`) и отображает подсказки или выполняет действия (в будущем).
    

## Базовый поток данных

Один цикл работы Windows‑клиента:

1. Убедиться, что клиент Raven 2 запущен и доступен.
    
2. Открыть окно аукциона и привести его к предсказуемому виду (размер/позиция окна, язык клиента и т.п.).
    
3. Сделать скриншот области экрана с таблицей лотов аукциона.
    
4. Прогнать скриншот через OCR и распарсить текст в список лотов.
    
5. Нормализовать имена террамондов в стабильные коды (`terramond_code`).
    
6. Сформировать JSON‑снимок по контракту `POST /snapshots`.
    
7. Отправить его на Linux‑backend и залогировать результат (успех/ошибка, `snapshot_id`).
    
8. (Опционально) запросить сигналы по интересующим террамондам и вывести/сохранить рекомендации.
    

## Логическая структура модулей Windows‑клиента

Проект Windows‑клиента (условно `raven2-windows-client`) логически состоит из следующих модулей:

1. **Launcher / Game control**
    
    - Назначение: запускать Raven 2 и управлять окном игры.
        
    - Функциональность:
        
        - запуск клиента, если он не запущен;
            
        - поиск окна Raven 2 по заголовку/классу;
            
        - приведение окна в нужное состояние (окно vs fullscreen, разрешение, расположение);
            
        - переход в меню аукциона (нажатие нужных клавиш/кнопок).
            
    - Интерфейс (пример функций):
        
        - `start_game()`
            
        - `ensure_game_running()`
            
        - `ensure_auction_open()`
            
2. **Snapshot capturer (захват скриншотов)**
    
    - Назначение: периодически делать скриншоты области экрана с аукционом.
        
    - Функциональность:
        
        - по расписанию (cron/loop) захватывать заданный прямоугольник экрана (область таблицы лотов);
            
        - сохранять PNG/JPEG‑файлы в локальную папку `screenshots/` с таймштампами в имени;
            
        - логировать путь к свежему скриншоту.
            
    - Интерфейс:
        
        - `capture_auction_screenshot() -> str` (возвращает путь к файлу).
            
3. **OCR + parser (распознавание и парсинг)**
    
    - Назначение: превратить скриншот аукциона в список лотов.
        
    - Этапы:
        
        1. OCR: вытащить текст из изображения (по строкам/ячейкам).
            
        2. Парсинг: по структуре UI аукциона выделить строки лотов и для каждой строки вытащить:
            
            - сырой текст имени террамонда;
                
            - цену за единицу/за стек;
                
            - количество;
                
            - направление сделки (`buy`/`sell`) — если UI это позволяет определить.
                
    - Внутренний формат (пример):
        
        json
        
        `[   {    "terramond_name_raw": "Basic Wood",    "price_raw": "123.45",    "quantity_raw": "100",    "direction_raw": "sell"  },  ... ]`
        
    - Интерфейс:
        
        - `parse_screenshot_to_raw_lots(path: str) -> list[dict]`
            
        - `normalize_raw_lot(raw_lot: dict) -> dict` (перевод в наш формат лота).
            
4. **Normalizer / catalog (словарь террамондов)**
    
    - Назначение: сопоставить «грязное» имя из OCR с устойчивым `terramond_code`.
        
    - Содержимое:
        
        - словарь (JSON/CSV/ini), где «варианты имён» → стабильный код и каноническое имя.
            
    - Пример:
        
        json
        
        `{   "Basic Wood": "wood_basic",  "BasicWood": "wood_basic",  "Базовое дерево": "wood_basic" }`
        
    - Интерфейс:
        
        - `normalize_terramond_name(ocr_name: str) -> (code: str, canonical_name: str)`
            
    - Результат нормализации используется при формировании полей `terramond_code` и `terramond_name` в JSON.
        
5. **Snapshot builder (формирование JSON под /snapshots)**
    
    - Назначение: собрать валидный JSON‑объект снимка аукциона для отправки в backend.
        
    - Структура, на которую ориентируемся (Linux‑backend уже это ожидает):
        
        json
        
        `{   "taken_at": "2026-03-08T10:00:00Z",  "source": "win_vm1_client1",  "lots": [    {      "terramond_code": "wood_basic",      "terramond_name": "Basic Wood",      "unit_price": 123.45,      "quantity": 100,      "direction": "sell"    }  ] }`
        
    - Интерфейс:
        
        - `build_snapshot(lots: list[dict], source: str) -> dict`
            
    - `taken_at` — время снимка (UTC или с таймзоной).
        
    - `source` — идентификатор клиента/VM (например, `"win_vm1_client1"`).
        
6. **HTTP‑клиент (общение с Linux‑backend)**
    
    - Назначение: отправлять данные на backend и, при необходимости, получать ответы (сигналы).
        
    - Функции:
        
        - `send_snapshot(snapshot: dict) -> snapshot_id | error` → `POST http://192.168.50.185:8000/snapshots`
            
        - (опционально) `get_simple_signal(code, current_price, direction_filter, days) -> dict` → `GET /signals/terramond/{code}/simple`
            
    - Особенности:
        
        - логирование HTTP‑кодов и ошибок;
            
        - базовый retry при временных сбоях сети.
            
7. **Scheduler / orchestrator (основной цикл)**
    
    - Назначение: организовывать периодический сбор данных.
        
    - Примерный псевдокод цикла:
        
        text
        
        `loop каждые N минут:   ensure_game_running()  ensure_auction_open()  path = capture_auction_screenshot()  raw_lots = parse_screenshot_to_raw_lots(path)  lots = [normalize_raw_lot(raw) for raw in raw_lots]  snapshot = build_snapshot(lots, source="win_vm1_client1")  send_snapshot(snapshot)  sleep N минут`
        
    - Плюс:
        
        - логирование всех этапов;
            
        - возможность ручного запуска «одного цикла» для отладки.
            
8. **(позже) Autoclicker / executor по сигналам**
    
    - Назначение: в автоматическом режиме исполнять сигналы (покупать/продавать), имитируя человеческое поведение.
        
    - Использует:
        
        - сигналы с backend’а (`/signals/...`);
            
        - функции управления окном игры и кликами.
            
    - Важные требования:
        
        - случайные задержки;
            
        - мелкие движения мышью;
            
        - ограничение времени сессий;
            
        - работа только на твинке на этапе тестов.
            

## Связь Windows‑клиента с Linux‑backend

Linux‑backend (`raven2-bot`, VM `bot-debian`) уже умеет:

- `POST /snapshots` — принимать снимки аукциона и записывать их в БД (`terramonds`, `snapshots`, `lots`).
    
- `GET /stats/terramond/{code}/basic` + `days`/`direction_filter` — выдавать статистику по цене и объёму за нужный период.
    
- `GET /stats/terramond/{code}/by_weekday_hour` — давать разрез по дням недели и часам.
    
- `GET /signals/terramond/{code}/simple` — выдавать простой сигнал (buy/sell/hold) относительно средней цены.
    
- `GET /prices/terramond/{code}/timeline` — отдавать таймлайн цен для построения графиков.
    

Windows‑слой должен:

- строго соблюдать формат JSON для `/snapshots`;
    
- по возможности использовать `source` для различения клиентов/окон;
    
- полагаться на backend для аналитики и сигналов, не дублируя сложную логику на Windows‑стороне.
    

## Минимальный первый прототип Windows‑клиента

Перед реальной интеграцией с игрой имеет смысл сделать прототип:

1. Вручную сохранить несколько скриншотов окна аукциона (без автокликера).
    
2. Написать небольшой скрипт (например, на Python), который:
    
    - читает эти скриншоты из папки;
        
    - гонит их через OCR;
        
    - парсит лоты;
        
    - собирает JSON по формату `/snapshots`;
        
    - отправляет его на backend `http://192.168.50.185:8000/snapshots`.
        
3. Проверить, что:
    
    - entries появляются в SQLite;
        
    - статистика `/stats/...` и таймлайн `/prices/...` корректно отражают распознанные данные.
        

После успешного прототипа можно переходить к:

- настройке Windows‑VM на Proxmox;
    
- автоматизации скриншотов и управления игрой;
    
- постепенному внедрению автокликера под руководством сигналов backend’а.
